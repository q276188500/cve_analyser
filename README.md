## 核心流程

### Step 1: 输入与数据获取

**输入**：
- 时间段（如：2024-01-01 至 2024-12-31）
- 或 CVE ID（如：CVE-2024-XXXX）
- 代码仓路径

**数据获取（cve-analyzer）**：
- CVE 漏洞描述
- CVE 漏洞关联的修复 patch
- CVE 漏洞关联的 Kconfig

### Step 2: 代码仓同步

- 每次评估前同步代码仓到最新
- `git fetch && git merge`

### Step 3: 初步筛选

**筛选逻辑**：

| 条件 | 结果 | 原因 |
|------|------|------|
| CVE 相关的 KCONFIG **未开启** | 无影响 | KCONFIG 未开启 |
| CVE 是**非内核问题** | 无影响 | 非内核问题 |

### Step 4: 详细评估

由 **OpenCLAW（内核专家）** 主导分析：

1. **获取 CVE 详情** - 从 NVD 获取完整信息
2. **查询代码仓** - 必须执行代码查询
3. **Patch 代码分析** - 读取实际代码分析
4. **知识库匹配** - 检索领域知识
5. **综合判断** - 给出合入建议

### Step 5: 报告生成

生成 Markdown 格式报告，包含：
- 基本信息
- 漏洞类型
- Kconfig 检查
- 影响评估
- 合入建议
- 内核专家详细分析

## 注意事项

1. **KCONFIG 筛选**：必须基于代码仓的 .config 进行筛选
2. **默认合入**：除非有充分理由，否则建议合入
3. **代码查询**：每个 CVE 必须读取实际代码
4. **独立分析**：每个 CVE 必须独立分析，不能批量处理
